# プロトタイプ検証: 結果と意思決定

> 入力: docs/03_prototype_plan.md, data/cards_draft.csv

## 前提の断り書き

本セッションでは物理的な人間プレイテストの実施者を確保できなかったため、
docs/03_prototype_plan.md のルールをそのまま数値シミュレーションに落とし込んだ
**簡易卓上シミュレーション（500ゲーム試行）**で代替した。CPU側は「固定ランダム
ダメージ（2〜4点/ターン）」という簡略化されたプレースホルダー行動しか取らせて
おらず、実際のCPUデッキ運用は未検証である。したがって本findingsは「数値バランスの
方向性」を掴むための一次情報であり、**実際の人間同士のプレイフィール（面白いか
どうか）の検証は依然として未実施**であることを明記する。実際の人によるプレイテストは
Phase6（テスト）のフィードバック取り込みプロンプトで改めて実施することを推奨する。

## シミュレーション結果

- 試行回数: 500ゲーム（弱点属性を4種均等に割り当て）
- プレイヤー勝率: 100%
- 決着までの平均ターン数: 2.9ターン（最小2／最大4）
- 攻撃属性の使用比率: 皮肉31.1% / 正論27.7% / 自虐からの逆転21.4% / 煽り19.8%

## 意思決定: 採用/廃止/修正

1. **【修正】誇りゲージとカードダメージの比率が破綻している**
   誇り20に対し、1ターン（コスト3）で10〜15点前後を削れてしまうため、
   2〜4ターンで決着する。docs/01_planning.md で定めた「1戦10〜15分」という
   セッション設計に対し、ターン数が少なすぎて体験として成立しない。
   → 誇りゲージの初期値を上げる（例: 30〜40）か、攻撃カードの基礎ダメージを
   現行の半分程度に下げる方向で、設計フェーズ（docs/04_balance_rules.md）で
   数値を再設計する。

2. **【保留】CPU側の防御的な行動選択が未検証**
   本シミュレーションではCPUがカードを使わず固定ダメージを返すだけの
   プレースホルダーだったため、「防御カードで受け流す」「弱点を突かれた側が
   耐える」といった読み合いの成立可否は判定できていない。
   → 設計フェーズでCPU AIの意思決定ロジック（最低限、弱点属性を隠しつつ
   時々防御カードを使う程度の簡易AI）を仕様化し、Phase6のシミュレーションで
   改めて検証する。

3. **【採用】属性相性システム（パターンA）はそのまま採用する**
   属性ごとの使用比率が19〜31%の範囲に収まっており、極端に選ばれない属性は
   発生していない。数値バランスの粗さはあるものの、メカニクスの骨格自体に
   構造的な欠陥は見当たらないため、パターンAを引き続き基本ルールとして採用する。

4. **【廃止判断なし】防御・強化カードの評価は次フェーズに持ち越し**
   本シミュレーションでは防御・強化カードを一切使用しない前提だったため、
   これらのカード種別の有用性は未評価。廃止するかどうかの判断はできず、
   設計フェーズでCPU AIとあわせて再検証する。

## 次フェーズ（設計）に引き継ぐ論点

- 誇りゲージ初期値、攻撃カードの基礎ダメージ、コスト対効果の基準値を
  数式化して docs/04_balance_rules.md に定義する（現行数値は暫定値であり
  そのまま実装しないこと）。
- CPU側の簡易AI仕様（弱点属性を完全に隠すか、ヒントとして部分開示するか）を
  設計フェーズで確定する。プレイヤーが「見抜く」快感を得るには、CPUの弱点情報の
  開示度合いが体験の核になるため、design.md で明示的に扱う。
- 防御・強化カードの数値バランスは、CPU AIの仕様確定後に改めてシミュレーションで
  検証する（本findings時点では未評価）。
- 誇り／怒りゲージの初期値変更に伴い、デッキ枚数上限（10〜20枚、要件定義での仮決定）も
  再検証が必要か設計フェーズで確認する。
