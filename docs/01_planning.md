# 企画: 罵倒デッキ構築カードゲーム

> 前提: 本ゲームはオリジナルキャラクター・オリジナル世界観で開発する。
> 実在の配信者・声優等の名前・容姿・音声・発言は一切模倣・引用しない。
> 「罵倒」というテーマ性・トーンのみを参照し、キャラクターは完全新規に設計する。

## 1. コアファンタジー

**プレイヤーは、場数を踏んだ「毒舌の論客」として、相手の弱点を的確に見抜く一言で
相手の心を折り、言葉の応酬の場を支配する快感を味わう。**

- 単なる罵り合いではなく、「相手のどこが弱点か」を読み、そこに刺さる言葉を
  デッキから選び抜いて当てる、という推理・読み合いの快感が中心。
- 殴り勝つのではなく「言い負かす」ことが勝利条件であることが、他の
  デッキ構築ゲームとの体験上の違いを生む。
- 「弱点を見抜く」手段: 対戦相手キャラクターの直近の言動ログ（例:
  「さっき自分のミスを他人のせいにした」等のバトル中に場に開示される
  行動履歴）から、性格タイプ（短気・プライド過剰・繊細・鈍感 等）を
  プレイヤー自身が推測し、対応する弱点属性の罵倒カードを選んで当てる。
  完全ランダムではなく「観察→推測→選択」の一連の思考プロセスを
  ゲームメカニクスとして可視化することで、「見抜く」快感を実際の
  プレイ体験に落とし込む。
- 想定ターゲット層: デッキ構築ゲーム（Slay the Spire等）の経験者を
  中核としつつ、対人の腹の探り合い自体を楽しめるライト層も許容する。
  年齢制限は設けないが、テーマ上「攻撃的な言葉」を扱うため、
  対象年齢・表現のトーンについては要件定義フェーズで改めて明文化する。

## 2. 差別化ポイント（vs Slay the Spire / Dominion 等）

1. **二重リソース制（誇りゲージ／怒りゲージ）**
   Slay the SpireはHPを削り合うだけの単一リソースだが、本作は
   「誇りゲージ（削られると敗北）」と「怒りゲージ（溜まると大技=決め台詞カードが解禁）」
   の2系統を持つ。罵倒が命中すると相手の誇りを削りつつ自分の怒りが上がるため、
   「今すぐ削るか、怒りを溜めて一発逆転を狙うか」というリスク管理の読み合いが生まれる。

2. **属性相性による会話の読み合い**
   罵倒には「煽り／正論／皮肉／自虐からの逆転」等の属性があり、対峙するキャラクターの
   性格タイプ（短気・プライド過剰・繊細・鈍感 等）ごとに弱点属性が異なる。
   Dominionのような「純粋な効率最適化」ではなく、「相手の性格を見抜いて弱点属性を
   選ぶ」というジャンケン的な対人読み合いの要素を加える。

3. **購入フェーズと直接対決フェーズの両立**
   Dominionは購入のみ、Slay the Spireは購入と戦闘が別ターンで完全に分離しているが、
   本作は「同じターン内で、デッキ強化（新しい罵倒カードの獲得）と直接対決を
   同時に選択できる」構造にし、テンポの速い１戦を成立させる。

## 3. プレイ時間・セッション設計

- **1戦（相手1人との対決）**: 10〜15分
- **セッション**: 3戦勝ち抜き（ローグライク的にランを進行し、勝つたびにデッキが強化される）で
  合計30〜45分程度
- 隙間時間に1戦だけ遊べる設計にしつつ、周回すると「育ったデッキで次の相手に勝つ」という
  デッキ構築ゲームらしい継続動機を持たせる。

## 4. 「罵倒」のメカニクスへの落とし込み（3パターン）

いずれもオリジナルキャラクター・オリジナル世界観を前提とする。

**パターンA: 属性相性システム**
罵倒カードに「煽り／正論／皮肉／自虐からの逆転」等の属性を持たせ、対戦相手のキャラクター
タイプ（短気・プライド過剰・繊細・鈍感）ごとに弱点属性が設定される。弱点属性が命中すると
ダメージ（誇り減少量）が増幅する。カードゲームとしての読み合いの核になる案。

**パターンB: 動揺ゲージ／ノイズカード混入**
罵倒が命中すると相手の「動揺ゲージ」が上昇し、一定値を超えると相手の次の手札に
「思考停止（何もできないカード）」「支離滅裂な反論（効果が半減するカード）」等の
ノイズカードが混入する。相手のデッキ運用そのものを妨害する案。

**パターンC: 連続罵倒コンボ／決め台詞デッキ**
同一ターン内で罵倒カードを連続して命中させると「連続罵倒ボーナス」が発生し、
怒りゲージが一定値に達すると使用可能になる「決め台詞カード（大ダメージ＋演出）」を
デッキに組み込める。コンボを組み立てる楽しさを罵倒というテーマに乗せる案。

> 3案は排他ではなく、A（属性相性）を基本ルールとし、B・Cを段階的に追加する
> 拡張パターンとして採用するのが有力（プロトタイプ検証フェーズで判断する）。

## 5. 想定プラットフォームと理由

**Web（ブラウザ）を第一候補とする。**

- ソロ開発・個人プロジェクトのため、インストール不要ですぐ遊べるWebが
  検証コスト・配布コストともに最小。
- React + TypeScript + Vite の構成であれば、後にPWA化やElectron化での
  デスクトップ配布への移行パスも残せる。
- モバイルはブラウザでの動作確認は行うが、タッチ操作最適化はMVPスコープ外とし、
  要件定義フェーズで明確に切り分ける。
