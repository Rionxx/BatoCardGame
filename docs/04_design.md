# 設計: 罵倒デッキ構築ゲーム

> 入力: docs/01_planning.md, docs/02_requirements.md, docs/03_prototype_findings.md
> 関連ファイル: data/cards_schema.json, docs/04_balance_rules.md

## 1. カードデータスキーマ

data/cards_schema.json に JSON Schema として定義済み。要点:

- カードは `type`（attack / defense / buff）、`attribute`（攻撃カードのみ、
  煽り／正論／皮肉／自虐からの逆転）、`cost`、`effects`（効果のリスト）、
  `flavor` を持つ。
- `effects` は type + value + duration の汎用構造とし、新しい効果種別を
  追加する際は enum に1件追加してコアロジック側にハンドラを1つ実装すれば良い。
  既存カードのJSON自体は変更不要（拡張性の担保）。
- 数値（ダメージ量等）はスキーマではなくdocs/04_balance_rules.mdの基準値に従い、
  data/cards.json（本番カードデータ、Phase5で作成）側に実数値として書く。

## 2. ゲームの状態遷移

```
[ゲーム開始]
    │
    ▼
[ラン開始: 誇り30/怒り0でリセット、対戦相手1体目を選出]
    │
    ▼
┌─────────────── 戦闘ループ（1戦） ───────────────┐
│                                                  │
│  [ターン開始]                                    │
│    │                                             │
│    ▼                                             │
│  [コスト回復] ── ターン1-2は+2、ターン3以降は+3(上限3) │
│    │                                             │
│    ▼                                             │
│  [手札補充] ── デッキから規定枚数ドロー              │
│    │                                             │
│    ▼                                             │
│  [プレイヤー行動フェーズ]                           │
│    ├─ カードを選択（コスト消費）                     │
│    ├─ 効果解決（ダメージ／防御／強化を即時適用）        │
│    └─ コストが尽きる or パスするまで繰り返し           │
│    │                                             │
│    ▼                                             │
│  [勝敗判定] ── 相手の誇りが0以下なら戦闘終了(勝利)     │
│    │ (継続の場合)                                  │
│    ▼                                             │
│  [CPU行動フェーズ] ── CPU AIがカードを選択・効果解決    │
│    │                                             │
│    ▼                                             │
│  [勝敗判定] ── 自分の誇りが0以下なら戦闘終了(敗北)     │
│    │ (継続の場合、ターン開始に戻る)                   │
│                                                  │
└──────────────────────────────────────────────┘
    │
    ▼
[戦闘勝利後: カード報酬選択(1〜3枚から1枚選びデッキに追加)]
    │
    ▼
[3戦勝ち抜きか？] ── No → 次の対戦相手を選出し戦闘ループへ戻る
    │ Yes
    ▼
[ラン終了（クリア）]
```

- 敗北時（自分の誇りが0以下）はラン終了（要件定義どおりセーブ機構なし、
  リザルト画面表示後にラン再開始のみ）。
- リソース処理順序: 「コスト回復 → 手札補充 → 行動 → 勝敗判定 → 相手ターン
  → 勝敗判定」の順で固定する。効果の持続時間（`duration`）はターン開始時に
  1ずつ減算し、0になった時点で効果を解除する。

## 3. カードパワーレベルの数値化ルール

docs/04_balance_rules.md として分離済み（実装後に数値のみ調整できるようにするため）。

## 4. モジュール分割・ディレクトリ構成案

```
romacoCardGame/
├── docs/                     # 各フェーズの成果物（本ファイル群）
├── data/
│   ├── cards_schema.json     # カードのJSON Schema定義
│   └── cards_draft.csv       # プロトタイプ検証用の暫定カードリスト
├── prototype/
│   └── print_cards.html      # 紙プロトタイプ印刷用HTML
└── src/
    ├── core/                 # ロジック層（UIに依存しない純粋なゲームルール）
    │   ├── data/
    │   │   └── cards.ts        # 本番カードデータ（cards_schema.jsonに準拠）
    │   ├── engine/
    │   │   ├── state.ts         # ゲーム状態の型定義（誇り/怒り/手札/デッキ等）
    │   │   ├── turn.ts          # ターン進行・コスト回復・手札補充の処理
    │   │   ├── effects.ts       # effects配列のtype毎のハンドラ実装
    │   │   └── resolveCard.ts   # カード使用→効果解決のエントリポイント
    │   ├── ai/
    │   │   └── cpuStrategy.ts   # CPUのカード選択ロジック
    │   └── run/
    │       └── runProgress.ts   # 3戦勝ち抜きのラン進行・報酬選択ロジック
    ├── components/            # UI層（React）
    │   ├── Battle/              # 戦闘画面（手札・誇り/怒りゲージ表示等）
    │   ├── CardReward/          # 勝利後のカード報酬選択画面
    │   └── Result/              # ラン終了リザルト画面
    ├── store/                 # Zustandによる状態管理（core/engineの状態をUIに橋渡し）
    └── App.tsx
```

- `core/` は React に依存しない純粋なTypeScriptとし、Vitestで単体テストしやすい
  構造にする（Phase6のロジックテスト・バランスシミュレーションの両方から
  同じ`core/`を再利用できるようにするため）。
- `store/` が `core/engine` の状態を保持し、`components/` はstore経由でのみ
  ゲーム状態を参照する（UIとロジックの責務分離）。

## 5. 拡張性・保守性の確認（自己チェック）

- 新カード追加: `data/cards_schema.json` を変更せずに `core/data/cards.ts` へ
  カードオブジェクトを1件追加するだけで対応できる（effectsの新種別が必要な場合のみ
  `core/engine/effects.ts` にハンドラを1件追加）。
- 数値調整: `docs/04_balance_rules.md` の基準値と `core/data/cards.ts` の実数値を
  変更するだけで、`core/engine` のロジックコードは変更不要。
- パターンB／C（動揺ゲージ、連続コンボ）を将来追加する場合、`core/engine/effects.ts`
  にハンドラを追加し、`GameState`（state.ts）に該当フィールドを追加する形で
  段階的に拡張できる設計とした。既存のパターンA実装への破壊的変更は不要。
